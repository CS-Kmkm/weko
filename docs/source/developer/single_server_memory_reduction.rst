単一サーバー継続時のメモリ削減余地
==================================

概要
----

本メモは、二サーバー構成を採らずに単一サーバーで WEKO3 を継続運用する前提で、
現行プログラムおよび同居ミドルウェアのメモリ削減余地を整理したものである。

対象は実行時メモリであり、Docker build 時のメモリ不足は別件とする。
また、本メモでいう「現行プログラム」は、WEKO3 の web/worker/scheduler に加えて、
同一ホストで動く PostgreSQL、Elasticsearch、Redis、RabbitMQ を含む運用スタック全体を指す。

結論
----

単一サーバーのままでメモリ削減できる余地はあるが、削減余地の大半は Python アプリ本体よりも
Elasticsearch、PostgreSQL、および一時的なバッチ処理の設定にある。

特に重要なのは次の点である。

* web は uWSGI ``processes = 1``、worker は Celery ``--concurrency=1 --prefetch-multiplier=1`` で、アプリ本体は既にかなり絞られている。
* 一方で、Elasticsearch は ``ES_JAVA_OPTS=-Xms512m -Xmx512m`` と固定 heap を持つため、単一サーバーでは最も分かりやすい常時消費メモリになりやすい。
* PostgreSQL も保守的な設定ではあるが、``max_connections`` やバッファ設定を実運用に合わせてさらに下げられる可能性がある。
* 再索引、bulk 更新、S3 転送などは定常時ではなくピーク時メモリを押し上げるため、チャンクサイズや並列度の見直し余地がある。

したがって、単一サーバー継続時の優先順位は「アプリの一般的な軽量化」よりも、
まずミドルウェア設定と重い運用処理の見直しを先に行うのが現実的である。

1万件規模での見立て
------------------

登録件数が 10000 件程度であれば、単一サーバーでのメモリ削減施策は依然として意味がある。
ただし、この規模では「件数そのものが大きいからメモリを食う」というより、
ミドルウェアの固定コストと、WEKO のメタデータ/検索マッピングの複雑さの影響が大きい。

件数 10000 件程度で見たときの判断は次のとおりである。

* PostgreSQL: 10000 件という件数だけでは、テーブルサイズ起因のメモリ圧迫は通常そこまで大きくない。効きやすいのは ``max_connections`` や buffer 類の見直しであり、件数依存の効果は限定的である。
* Elasticsearch: 件数 10000 件は Elasticsearch としては大きくないが、WEKO の mapping は ``index.mapping.total_fields.limit = 50000``、多数の ``copy_to``、``nested``、``fielddata`` を含んでおり、文書件数だけでなくメタデータの形がメモリに効く。そのため、Elasticsearch heap の見直しはこの規模でも効果が出やすい。
* web / worker: 既に低並列設定なので、10000 件規模でも大きな削減余地は小さい。
* 再索引や bulk 更新: 10000 件であれば処理時間がまだ現実的な範囲に収まりやすく、``chunk_size`` を下げてピークメモリを削る施策を取りやすい。

要するに、10000 件規模では「二サーバー化しないと厳しい」とまでは言いにくい一方で、
Elasticsearch heap、PostgreSQL 接続設定、bulk/reindex のチューニングはまだ十分に意味がある。
特に Elasticsearch は固定コスト寄りの成分が大きいため、件数が 10000 件でも見直し候補から外れない。

現行設定から見える事実
----------------------

web / worker / scheduler
^^^^^^^^^^^^^^^^^^^^^^^^

現行設定では、アプリ本体はすでに比較的低メモリ寄りである。

* uWSGI は ``processes = 1``、``threads = 2`` で起動している。
* uWSGI には ``reload-on-rss = 256`` と ``max-requests = 100`` があり、メモリ膨張をある程度抑える方向になっている。
* Celery worker は ``--concurrency=1 --prefetch-multiplier=1`` で起動している。
* Celery beat は worker とは別プロセスで動いている。

このため、web/worker 側だけで大幅な常時メモリ削減を得る余地は大きくない。

PostgreSQL
^^^^^^^^^^

PostgreSQL の現行設定は次のとおりである。

* ``shared_buffers=128MB``
* ``work_mem=4MB``
* ``maintenance_work_mem=64MB``
* ``temp_buffers=8MB``
* ``max_connections=40``

全体としてはかなり控えめな部類だが、実利用ユーザ数や同時接続数が少ないなら、
``max_connections`` と一部バッファ設定をさらに絞れる可能性がある。

なお、``effective_cache_size=512MB`` は planner hint であり、実メモリをそのまま確保する値ではない。
この値を下げても大きな実メモリ削減にはつながらない。

Elasticsearch
^^^^^^^^^^^^^

Elasticsearch は次の設定で動作している。

* Elasticsearch 6.8.23
* single-node 構成
* ``ES_JAVA_OPTS=-Xms512m -Xmx512m``

この 512 MiB heap は単一サーバーでは目立つ固定コストであり、メモリ削減の最重要候補である。
ただし、検索件数、インデックスサイズ、再索引頻度によっては下げ過ぎると検索遅延や GC 増大、
indexing failure の原因になりうる。

また、WEKO の Elasticsearch mapping は 1 shard 前提である一方、フィールド数上限が大きく、
``copy_to``、``nested``、``fielddata`` を多用する。したがって、件数が 10000 件程度でも
「単純な 10000 文書検索エンジン」よりはメモリ要求が高くなりやすい。

Redis / RabbitMQ
^^^^^^^^^^^^^^^^

* Redis には Compose 上で ``memory: 100m`` の limit がある。
* RabbitMQ には明示的な memory limit はないが、現行構成では大きな並列処理をかけない前提になっている。

単一サーバーの支配的メモリ要因としては、通常は PostgreSQL と Elasticsearch の方が大きい。

削減余地の評価
--------------

優先度: 高
^^^^^^^^^^

1. Elasticsearch heap の再評価

* 効果: 常時メモリ削減に最も効きやすい。
* 現行値: ``-Xms512m -Xmx512m``
* 検討内容: 10000 件規模で、かつメタデータ量が極端に大きくないなら 384 MiB や 256 MiB まで下げられる可能性がある。
* 注意: 検索応答、GC、再索引、集約処理が悪化しやすいため、最初に測定してから段階的に下げるべきである。

2. PostgreSQL の同時接続数とバッファ設定の再評価

* 効果: 常時メモリとピーク時メモリの両方に効く可能性がある。
* 主な対象: ``max_connections``、``shared_buffers``、``maintenance_work_mem``、``temp_buffers``
* 検討内容: 実際の接続数が小さいなら ``max_connections`` を 40 から 10-20 程度へ下げる候補がある。
* 注意: ``work_mem`` を下げるとソートや集約で一時ファイルが増え、性能低下を招きやすい。

3. 不要な重い保守処理の実行頻度見直し

* 効果: 常時ではなくピーク時メモリと CPU 使用率の平準化に効く。
* 主な対象: 再索引、著者一括更新、統計集計、ファイル変換系の定期処理。
* 検討内容: 実行時間帯を夜間へ寄せる、メンテナンス時のみ手動実行にする、頻度を下げる。

優先度: 中
^^^^^^^^^^

4. bulk index / 再索引のチャンクサイズ縮小

* 効果: 再索引や大量更新時のピークメモリを下げやすい。
* 現行値: ``chunk_size=500``、``max_chunk_bytes=104857600``、``INDEXER_BULK_REQUEST_TIMEOUT=600``
* 検討内容: 10000 件規模なら、たとえば ``chunk_size`` を 100-200 に下げても、処理時間の増加を許容しやすい可能性がある。
* 注意: 所要時間は伸びやすく、これは定常運用ではなく保守時のチューニングである。

5. S3 転送並列度の低減

* 効果: S3 互換ストレージ利用時の一時メモリ削減に効く。
* 現行値: ``WEKO_RECORDS_UI_S3_TRANSFER_USE_THREADS = True``、``WEKO_RECORDS_UI_S3_TRANSFER_MAX_CONCURRENCY = 10``
* 検討内容: 2-4 程度へ下げる、必要なら thread を無効化する。
* 注意: S3 を使っていない環境では効果がない。

6. 検索レスポンス整形時の並列度見直し

* 効果: 検索結果の整形時に発生する一時的な thread / object 使用量を抑えられる可能性がある。
* 現行コード: ``ThreadPoolExecutor(max_workers=10)``
* 検討内容: 2-4 程度へ下げる。
* 注意: これはコード変更であり、検索レスポンス時間への影響確認が必要である。

優先度: 低
^^^^^^^^^^

7. uWSGI thread 数の削減

* 効果: web 側の常時メモリをわずかに下げられる可能性がある。
* 現行値: ``processes = 1``、``threads = 2``
* 検討内容: ``threads = 1`` を試す。
* 注意: 既に ``processes = 1`` まで絞っているため、削減幅は小さい。応答性低下の方が目立つ可能性がある。

8. Celery beat の別プロセス運用見直し

* 効果: Python プロセス 1 本分の削減余地がある。
* 検討内容: worker と beat の統合、または外部 scheduler への移行。
* 注意: 運用信頼性とのトレードオフが大きく、安易には推奨しにくい。

すでに余地が小さい箇所
----------------------

次の項目は、現行設定を見る限り既にかなり抑えられており、大きな削減余地は小さい。

* web の uWSGI process 数はすでに 1。
* Celery worker の concurrency はすでに 1。
* prefetch multiplier も 1。
* PostgreSQL の ``shared_buffers=128MB`` はすでに小さめ。
* Redis limit は 100 MiB に抑えられている。

そのため、「アプリ本体のプロセス数を減らす」方向で得られる効果は限定的である。
大きな効果を狙うなら、Elasticsearch heap と重い保守処理の見直しを優先すべきである。

10000 件規模であれば、この傾向はさらに強い。件数が比較的小さいため、アプリ本体の常駐メモリよりも、
ミドルウェアの固定コストをどう下げるかの方が効きやすい。

実行前に確認すべきこと
----------------------

設定を下げる前に、少なくとも次を取得しておくべきである。

#. コンテナ単位の実メモリ使用量。
#. web / worker / scheduler の RSS 推移。
#. PostgreSQL の同時接続数と最大接続数。
#. Elasticsearch の heap 使用率、GC、検索応答、再索引時間。
#. どの操作で OOM や swap が起きるか。
#. バッチ実行時だけ増えるのか、常時高いのか。

判断の目安は次のとおりである。

* 常時メモリが高い: Elasticsearch heap、PostgreSQL、常駐プロセス構成を優先して見直す。
* 保守時だけ高い: reindex、bulk、S3 転送、集計系のチャンクサイズや並列度を優先して見直す。
* web だけ高い: uWSGI thread 数や検索レスポンス整形処理の見直し候補を調べる。

10000 件規模で OOM や swap が起きるなら、件数の絶対量よりも、次のどれかを疑うべきである。

* Elasticsearch heap が単一サーバーに対して大きすぎる。
* 再索引や一括更新のチャンクサイズが大きい。
* 1 件あたりのメタデータが大きく、検索インデックスが重い。
* ファイル処理や S3 転送が一時的にメモリを押し上げている。

推奨アプローチ
--------------

単一サーバー継続時は、次の順序で進めるのが妥当である。

1. 実測で、どのコンポーネントが最もメモリを使っているか確認する。
2. Elasticsearch heap を小さく試す。
3. PostgreSQL の ``max_connections`` とバッファ設定を、実利用に合わせて下げる。
4. 再索引や bulk 処理のチャンクサイズを下げ、ピークメモリを抑える。
5. 必要な場合のみ、uWSGI thread 数や検索整形コードの並列度を見直す。

この順序であれば、リスクの大きいコード変更を避けながら、効果の大きい箇所から順に調整できる。
